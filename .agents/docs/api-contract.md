## Experimental stability policy

Here, “public contract” means the authoring surface intentionally supported, documented, typed, and tested in the current version. It does not promise cross-release backward compatibility while vue-tui remains experimental. Under the [vouched product policy](./goal.md#api-stability-during-experimentation), a public value, type, option, path, or behavior may be renamed, moved, changed, or removed directly when an accepted target design supports that change; aliases, deprecation windows, and runtime shims are not required merely because an API shipped before 1.0.

The public-surface guards remain important. They make every change deliberate, prevent accidental exports or type drift, and prove that the resulting current API is internally coherent. A clean-slate change updates the implementation, value and type guards, documentation, examples, tests, and first-party consumers together.

Historical F1.8 applied that process to `useRenderSession()` and `useLayoutSize()`, proving that the implementation's session graph was internally coherent. The active Path 1 re-audit found that applications need only the root layout width and, separately, the presence and value of a finite visual height. The current public surface therefore exports `useLayoutWidth(): Readonly<Ref<number>>` and `useViewportHeight(): Readonly<Ref<number>> | null`; it removes `useRenderSession()`, `useLayoutSize()`, `RenderModeResolution`, `RenderOutput`, `RenderSession`, `RenderSize`, `RenderLayoutSize`, and `UseLayoutSizeReturn` rather than preserving aliases. The broader session resolver remains necessary internal Runtime state. `RenderMode` remains public because mount options use it. The older F1 completion evidence is implementation history, not a reason to restore the broad projection.

F3 applied the same process to input. `useInput()` now receives a frozen `TuiInputEvent` key, text, paste, or uninterpreted union and requires an `InputHandlerResult`; `useInputAvailability()` publishes one stable readonly host capability. The guarded public types include `TuiInputEvent`, `TuiInputSource`, `TuiInputModifiers`, `TuiInputPhase`, `InputHandler`, `InputHandlerResult`, `InputRouteDecision`, `InputAvailability`, `UseInputOptions`, and `UseInputAvailabilityReturn`. `usePaste`, public `Key`, public raw-mode controls, mount and testing `rawMode`, and `exitOnCtrlC` were removed directly, while `useStdin().stdin` remains the exact raw mounted-stream escape hatch. Repository, public/type, clean-consumer, host, lifecycle, HMR, PTY, visual, restoration, CI, and independent-review gates agree with that surface, so F3 is complete.

F4 replaced string focus IDs, manager lookup and enable/disable calls, setup-order Tab traversal, automatic Escape blur, and per-focus raw-input ownership with six root values—`useFocus`, `useFocusScope`, `useFocusedInput`, `useFocusScopeInput`, `useExternalInput`, and `useFocusManager`—and seven named types—`UseFocusOptions`, `UseFocusReturn`, `UseFocusScopeOptions`, `UseFocusScopeReturn`, `ExternalInputHandler`, `ExternalInputSource`, and `UseFocusManagerReturn`. Opaque ref-bound handles, rendered preorder, active and trapped scopes, centralized restoration, and selected F3 routing are complete. After an authoritative renderer commit, host-only rendered reconciliation reuses the accepted focus generation only when no logical mutation is pending and the route boundary, owner, ordered ancestors, external owner, full rendered-target sequence, and eligible sequential-target sequence are identity- and order-equal. Any logical mutation or route/traversal change publishes a replacement generation; reuse preserves the demand lease. F3 then fails only the stale selected path closed, while independently captured application-global handlers for the same fact remain unaffected. The final focus-generation correction changes no public value or named type.

Historical F5 proved a sound accepted-paint geometry service, rendered-target lifetime, and caret transport, but its public `useElementGeometry()` projection exposed more than real layout consumers used. Path 1 now publishes only `useBoxSize()` and `BoxSize`: the target must be a direct ref to the exported `Box` in the current app, and the readonly ref contains either the last accepted full `{ width, height }` or `null`. A non-Box or foreign-app target throws. `CellRect`, `ElementGeometryFragment`, `ElementGeometry`, and `UseElementGeometryReturn` leave the public contract; rich fragments, clipping provenance, surface coordinates, and target tracking remain internal. `CellPoint`, `ElementTarget`, and the current caret types remain public only while Paths 4 and 6 audit the narrower caret and mouse contracts; Box measurement does not justify them. The older F5 completion evidence remains evidence for the retained mechanisms rather than the broad authoring shape.

F6 applied the same process to targeted mouse input. `@vue-tui/runtime/fullscreen`, the first supported public subpath, initially exposed `useMouseEvent()` and `useMouseDrag()` plus the selected named mouse types; it references rather than re-exports root `CellPoint` and `ElementTarget`, and does not duplicate `createApp()`, common components, focus, geometry, or render-session facts. Completed F8 deliberately adds `useTextSelection()` and its named types to that same supported subpath, so its current exact value set is `useMouseEvent`, `useMouseDrag`, and `useTextSelection`. `@vue-tui/testing` exposes parsed physical mouse phases, modeled reporting state, and the F8 clipboard model so public tests exercise production click, drag, selection, and copy paths. Package exports, declarations, exact value/type guards, template/TSX/JavaScript rejection, clean Vue 3.4/TypeScript 6 consumption, and the direct native macOS Terminal selection/wheel observation agree with the implemented surfaces. The exact pointer behavior and direct v1 removals are recorded in [targeted-pointer.md](./targeted-pointer.md); the final subpath addition and selection contract are recorded in [fullscreen-selection-and-copy.md](./fullscreen-selection-and-copy.md).

F7 applied the same process to the existing `@vue-tui/components` `ScrollBoxExpose` handle without adding an export. `scrollToLine()`, `scrollByLines()`, `scrollToTop()`, and `scrollToBottom()` retain their names and parameters and replace `void` directly with `boolean`; `true` means only that the effective top line changed after flooring and clamping. A no-move `scrollToBottom()` still re-arms sticky following and returns `false`. The public wrapper does not expose the private sticky control, and non-finite JavaScript inputs fail before state mutation. Named-handle and component-instance types, SFC/TSX consumers, packed JavaScript behavior, clean Vue 3.4.38/TypeScript 6.0.3 consumption, deterministic nested focus/pointer routing, real PTYs, visual restoration, and the historical repository and CI gates agree, so F7 is complete. The exact behavior and closure evidence are recorded in [ScrollBox](./components/scroll-box.md#f7-accepted-contract-and-closure-evidence).

F8 supplies one common clipboard service and one Fullscreen semantic Text selection service. The runtime root adds `useClipboard()` plus `ClipboardTransport`, `CustomClipboardTransport`, `Osc52ClipboardTransport`, `ClipboardTransportResult`, `ClipboardAvailability`, `ClipboardUnavailableReason`, `ClipboardWriteResult`, and `UseClipboardReturn`; `MountOptions.clipboard` accepts exactly one explicit custom or OSC 52 transport. `@vue-tui/runtime/fullscreen` adds `useTextSelection()` plus `UseTextSelectionOptions`, `TextSelectionCommands`, `TextSelectionCopyResult`, `TextSelectionMove`, `TextSelectionRange`, `TextSelectionState`, and `TextSelectionUnavailableReason` without duplicating the root clipboard types. `@vue-tui/testing` adds `TestClipboardBehavior`, `TestHost.clipboard`, and readonly `RenderResult.clipboard.requests`. Package, HMR, PTY, image-observed visual, native custom-clipboard, restoration, historical repository and CI, and independent-review evidence agree at the F8 checkpoint; final local Runtime-closure evidence reconfirms the contract without claiming a new remote run. F8 is Done. The exact behavior and closure evidence live in [Fullscreen text selection and clipboard](./fullscreen-selection-and-copy.md).

The active Runtime-foundation re-audit reduces terminal history to one `Static` value on the supported `@vue-tui/runtime/inline` entry. It has no props, collection-specific named types, or scoped item/index payload. One mounted instance commits one ordinary slot tree once; Vue `v-for` and stable keys own collection identity. Runtime owns the irreversible output transaction: output-free success accepts, a normally returned backpressured write accepts exactly once, and a throwing write abandons the whole instance without retry. Effective visual Fullscreen rejects component presence, while effective Inline, Fullscreen-to-screen-reader fallback, non-TTY streams, deterministic modeled non-Fullscreen hosts, and string rendering retain history behavior. The previous `items`, `style`, and `Object.is` committed-prefix contract remains historical implementation evidence rather than the current target.

The same re-audit reduces the common rendering vocabulary to the irreducible `Box` and `Text` primitives. `BoxProps` has exactly 28 accepted fields and `TextProps` exactly eight, as listed in [the active re-audit](./runtime-public-foundation-reaudit.md#box-and-text-prop-family-audit); their render-time validators reject invalid structure before host mutation and enforce the small public color grammar on visual documents. The public `Color` type names that grammar, with Runtime enforcing the exact six-digit hex rule that TypeScript cannot express economically. `Newline`, `Spacer`, `Transform`, `useAnimation`, `BoxStyle`, and `BoxLayoutStyle` are not common-root contracts. Newlines and flex spacers are ordinary Text/Box composition, Spinner owns its own timer, and the proven Transform renderer remains private implementation material. Box and Text disable Vue attribute inheritance and reject every remaining `$attrs` key before host creation: removed mouse listeners receive a Fullscreen-composable diagnostic, while stale props, typos, and browser-only attributes receive the ordinary undeclared-attribute error. Vue's `key`, `ref`, and vnode lifecycle hooks remain component mechanics rather than `$attrs`.

Runtime closure also changed coordinated output from an unobservable `void` side effect to the guarded `CoordinatedWriteResult` union. `accepted` with `writable: true` permits another transaction immediately; `accepted` with `writable: false` means the bytes were handed exactly once and supplies `ready`; `blocked` means the body was not invoked and no caller bytes were retained, also supplying the current gate's `ready`. The Runtime root exports the named type, and `UseStdoutReturn.write` plus `UseStderrReturn.write` use it exactly. Raw streams remain explicit uncoordinated escape hatches. Slow-writer capacity and release evidence passes.

Runtime closure implements Vue's built-in `v-show` for a Box-rooted subtree through the minimal host `style.display` contract that runtime-dom invokes directly. The host composes authored Box `display` with the directive's temporary hidden state: `v-show=false` always hides, while showing again applies the latest authored value and therefore does not override `display=none`. Hidden state remains mounted and reactive but is absent from layout, paint, focus eligibility, visible geometry, caret ownership, and Fullscreen hit testing. Direct `v-show` on public `Text` or `Static` is not part of this contract because their host shape or internal Yoga role differs. Focused template, Inline/Fullscreen, real-PTY, visual, and clean packed Vue 3.4 evidence passes.

Runtime closure also accepts PR #251's public behavior without copying its unsafe raw-marker mechanism. Nested `Text` with `color="revert"` or `color="initial"` resets only foreground to the terminal default, survives wrapping and further nested resets, and lets the enclosing foreground resume afterward. Background and retained boolean styles continue to compose. Runtime carries reset boundaries as private structural chunks rather than reserving Unicode characters, so literal private-use text and private Transform output remain content. Focused, final-tarball Vue 3.4, visual, and repository evidence passes. The complete final evidence for these closure surfaces is centralized in [Runtime foundation closure](./runtime-foundation-closure.md#final-closure-evidence).

## The contract = exports from supported package entry points **and their user-consumable types**

The public API is everything exported from the common root (`@vue-tui/runtime`) and every explicitly documented supported public subpath, together with **their types**: component prop types, composable return/options types, and named types such as `AriaRole`, `BoxSize`, `BoxProps`, `UseXReturn`, and `UseXOptions`. A package `exports` entry is not sufficient by itself; the path becomes supported only when the project documents and guards it as an authoring surface.

A type is **as much a part of the current contract as runtime behavior**. If user code can name a type and annotate with it, changing or removing it changes the supported authoring surface at compile time. That is allowed during experimentation when deliberate, but the type surface must be designed, updated, and tested with the same care as runtime behavior.

Because it is contract, it is **tested, not merely shipped**:

- `public-api.test.ts` snapshots the **exact** common-root value-export set — adding, removing, or
  renaming any runtime export fails it, so every surface change must be a deliberate edit there. Each supported public subpath needs its own exact value-export guard, named-type checks, declaration inspection, and clean package consumer.
  Specific internal-only members (`measureText`, the screen-reader linearizer) are additionally
  tripwired, and internal-only _types_ (e.g. `ScreenReaderOptions`) are guarded with a compile-time
  `@ts-expect-error`. Type-only exports are erased at runtime, so the _type_ surface is guarded
  individually rather than exhaustively snapshotted.
- Type-_safety_ behavior is established by RUNNING the type-checker against real usage (`tsc` for
  TSX, `vue-tsc` for templates), never assumed. See [accessibility-api](./accessibility-api.md) for a worked example —
  which aria spellings the compiler does and does not catch, proven with both tools.

## `/internal` is NOT the contract

`@vue-tui/runtime/internal` is an explicitly internal / advanced surface — host-node types (`TuiNode`, …), test-only helpers (`renderToStringWithScreenReader`), dev/HMR types (`DevState`, `DevErrorInfo`), kitty-controller internals, etc. It is **not a supported authoring surface**: its exports are not documented or guarded as a complete set by `public-api.test.ts` (a member may be tripwired there to prove it is internal), and may change without the deliberate public-surface process above.

Placement rule for any export:

- A user-facing contract whose semantics are common across supported rendering surfaces → the **main barrel**.
- A user-facing contract that intentionally requires one explicit terminal surface or integration boundary → a documented **public subpath**, when that boundary is part of the selected design rather than a directory-organizing convenience.
- Needed only by tests or advanced integrators → **`/internal`**, never the main barrel.

Packaging/build internals (the `exports` field shape, `.mjs` paths, `dist` layout) are likewise
**not** part of the behavioral/type contract and are not aligned to Ink — see the alignment-scope
note in [ink-divergences](./ink-divergences.md).

### Accepted incidental exposure: `TuiNode` via `TuiApp`

`TuiNode` is an `/internal` type, but it is **incidentally reachable** through the public `TuiApp`, which `extends Omit<App<TuiNode>, "mount">` to inherit Vue's full app surface — Vue's `App<HostElement>` carries the host type on its internal `_container` field. This is a **conscious non-fix, not a supported authoring contract**: `_container` is a Vue-internal field no consumer uses, so the exposure is cosmetic. Narrowing it (`App<unknown>` / a `Pick<App, …>` allowlist) was considered and skipped as ceremony without user-visible benefit. Treat `TuiNode`-through-`TuiApp` as unsupported and don't re-flag it.
