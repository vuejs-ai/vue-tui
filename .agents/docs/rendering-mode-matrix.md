# Rendering mode and host behavior matrix

> **Status:** current visual host-behavior evidence with a historical F1 public-projection layer. The supported public live-host selector is only `MountOptions.mode`; output cadence, render observation, incremental painting, pointer, selection, and clipboard passages below describe private implementation or historical candidate evidence rather than mount options or public hooks. The internal surface resolver, Inline invariant, application-side escape-hatch requirement, and finite clean-slate mount behavior remain applicable. Screen-reader presentation and every public or private selector for it are removed.

## Why the matrix has more than two rows

Inline and full-screen are the two live terminal screen models. They are not the only rendering surfaces vue-tui currently produces. A redirected stream, `renderToString`, and deterministic observation do not acquire the same surface merely because an application or test requested a mode.

The matrix therefore keeps four questions separate:

1. **Render host:** live process, deterministic test, or synchronous string render.
2. **Requested mode:** inline or full-screen, when the host accepts a terminal-mode request.
3. **Effective surface:** main-screen live region, alternate-screen viewport, final stream output, static string, or test observation.
4. **Independent I/O facts:** Runtime's private output-cadence policy, stdout and stdin TTY state, and terminal protocol availability.

This matrix is the source behavior model rather than a public type definition. The internal resolver still maps these axes into one coherent session, but applications do not receive that whole graph. The current Path 1 projection exposes root width on every host and an optional bounded visual height.

## Current public resolution and private surface mechanics

The live renderer now validates and resolves one surface before reserving stdout or mutating terminal state. [`render-session.ts`](../../packages/runtime/src/render-session.ts) owns the pure resolution; [`render.ts`](../../packages/runtime/src/render.ts) consumes its result:

```ts
requestedMode = normalize(publicOptions.mode ?? "inline");
liveUpdates = derivePrivateOutputCadence(stdout);
dimensions = resolvePositiveStdoutSizeOrProvenanceAwareTerminalProbe();
surface = resolveLiveSurface({ requestedMode, liveUpdates, stdout, dimensions });
alternateScreen = surface.kind === "fullscreen-terminal";
elementHitTesting = surface.kind === "fullscreen-terminal";
```

The retained private F6 mechanism does not resolve or acquire a blanket mouse path here. Its hit-testing, reporting, and capture code is not reachable through a supported `/fullscreen` entry or public pointer hook; it remains evidence for a future narrower proposal.

The retained private F8 mechanism likewise does not add a surface resolver input. Its Text mapping, selection controller, and clipboard service remain internal evidence; Runtime currently exposes no selection hook, clipboard hook, clipboard mount option, or OSC 52 operation.

Box-rooted `v-show` also does not add a surface-resolution branch. Vue's directive writes the Box host's minimal `style.display` facade, which composes with the latest authored Box `display` value and updates Yoga. Both visual modes then derive layout, paint, focus eligibility, geometry, caret, and Fullscreen hit testing from the same effective `DISPLAY_NONE` state. The subtree remains mounted and reactive. This host seam is intentionally not installed on `Text`, context-dependent virtual Text, `Transform`, or `Static`.

The supported `mode` option accepts only `"inline"`, `"fullscreen"`, or `undefined`; omission becomes an Inline request. Own `fullscreen`, `alternateScreen`, `interactive`, or `debug` keys fail synchronously, including when their value is `undefined`. At the 2026-07-24 audit checkpoint, the branch still recognized the repository-only string keys `liveUpdates`, `onRender`, `maxFps`, `incrementalRendering`, and `clipboard` through public `app.mount()`; that was an implementation leak, not a public contract. The current implementation associates retained repository-only controls with an otherwise ordinary public-options object through module-private state, and the public testing bridge invokes the original Runtime-owned mount path without exposing that association to a caller-owned `mount` wrapper. Every hidden or superseded string is rejected before stream getters, stdout ownership, terminal probing, raw mode, listeners, Vue setup, or terminal writes.

The private `liveUpdates` fact controls output cadence and does not state whether the application can receive input. Runtime derives normal public-mount cadence from the selected host and records requested and effective mode, output, dimensions, and structural capabilities in its private coherent session. Public components receive only the width and optional bounded height they can act on; `useInput()` and `useStdin()` provide the separately reviewed input contracts.

Terminal dimensions carry provenance. A complete positive stdout pair wins. If it is incomplete, one complete positive pair from process stdout/stderr, `COLUMNS`/`LINES`, a controlling-terminal ioctl, `tput`, or Linux `resize` replaces it atomically; fields from two sources are never spliced into a claimed viewport. Process-global fallback is used only for the process's own stdout/stderr, not an arbitrary custom TTY. Probe failure returns unavailable instead of manufacturing `80×24`. A valid partial width may still inform an unbounded layout, but it cannot acquire a visual terminal mode. The renderer may choose 80 layout columns, but that default never establishes terminal dimensions; `useViewportHeight()` returns `null` whenever the effective layout is unbounded.

The supported public mount derives its row from stdout capabilities, usable dimensions, and requested mode: ordinary TTY output maps to L4-L6 and non-TTY output maps to L1. Internal test and repository mechanisms can still exercise L2 and forced L1 directly, but that control is not a public mount option. No supported option adds another presentation axis.

`Static` is the accepted `@vue-tui/runtime/inline` terminal-history primitive. One mounted instance commits one ordinary slot tree once; Vue iteration and stable keys own collections, and remounting begins a separate history block. An output-free eligible transaction leaves the instance open, while an indeterminate throwing write abandons it without retry. On L6, rejection happens before Static bytes, dynamic output, observation, or a new frame. The branch's more exact live ordering, hidden-ancestor behavior, placement, nesting, and Static-specific failure timing remain implementation evidence under review.

### Observation and scheduling

Deterministic commit observation uses an internal render observer that receives structured `{ dynamic, staticOutput }` content independently from terminal bytes. Installing the observer does not select a host, change update cadence, append frames to stdout, or manufacture a capability. Internal tests may select unthrottled scheduling, but `maxFps` is not a supported public mount option. The former `debug` mount option is removed and fails synchronously before stream inspection or terminal mutation; `@vue-tui/testing` exposes supported observation through `RenderResult` instead.

### Development-reload modifier

An in-place Vue hot update keeps the mounted session and repaints through its existing surface. A Vite full reload runs the registered internal teardown before replacing the app, restoring terminal modes and the alternate screen without settling the old app's exit promise; the newly imported app then acquires its surface again. The target keeps that distinction: a hot update preserves the session, while a full reload must completely release and reacquire terminal ownership.

Mount acquisition, suspension, continuation, repaint, and teardown are exception-safe lifecycle transactions. Listener registration precedes terminal acquisition; a partial mount rolls back every resource acquired so far. Normal unmount, app exit, process exit, `beforeExit`, SIGINT, SIGTERM, SIGHUP, Runtime failure, and full HMR reload release only owned raw mode, bracketed paste, mouse level, Kitty keyboard state, cursor state, alternate screen, listeners, and stream reservation. Cleanup continues when one release throws. An ordinary teardown or settlement re-entered from a nested stream write is deferred until the current acquisition or repaint has finished and every resource can receive its release turn. A non-returning `process.exit()` or signal-exit callback is the exception: it performs immediate synchronous restoration, skips final user rendering and Vue lifecycle hooks, and returns control to Node only after owned terminal state has been released. A second live app on the same stdout throws synchronously before mutation and may retry after the owner releases it. Direct process-stream writes bypass coordination; default console protection uses one global patch whose active-app stack routes each intercepted record through the current owner's coordinated output.

## Historical F1 implementation record

> **Historical boundary:** everything from this heading to the end of the file records the earlier F1 design and its then-current tests. It is retained as implementation evidence, not as a current API or hidden feature contract. In particular, its screen-reader rows, `presentation` facts, transcript helpers, and linked tests were superseded and removed on 2026-07-22; current behavior is defined by the sections above and the [Runtime public API decision ledger](./runtime-public-api-decisions.md).

### F1.2 accepted mount contract at the time

F1.2 settles one canonical `createApp` and one optional public mode field, `mode: "inline" | "fullscreen"`. Omission is an Inline request for convenience; it does not make Inline the primary product mode or Fullscreen a degraded path. The old mode booleans are removed directly because the project is experimental; F1.4 also replaces the broad `interactive` output override with `liveUpdates`.

On a TTY, an explicit live-update override also overrides the CI default and resolves through the corresponding TTY row when terminal dimensions are usable. The live resolver first accepts a positive stdout size, then tries the provenance-aware controlling-terminal probe. A positive detected width may still supply layout width, and 80 is the final layout-column default, but that default is not labeled a detected terminal without a positive sourced pair and unbounded layout rows remain `null`. Visual mode acquisition without a coherent terminal pair falls back to final stream output. The override changes output cadence, not terminal capabilities: it cannot acquire Fullscreen, an alternate screen, stable viewport, or element hit testing on a non-TTY stream. Deterministic observation is supplied separately by the internal observer and therefore cannot change any row. Likewise, stdin and raw-input behavior remain independent resolver inputs rather than consequences of the public output facts.

The test host exposes structured content frames and an emulated final terminal surface as different observations. A final-stream test uses the production final-output path, while an explicitly live stream test uses the production updater without claiming a terminal mode. Static and test dimensions are supplied deliberately by their hosts and never borrowed from the developer's process streams.

Fallback only describes an environment in which the requested screen model cannot exist. It does not make Inline primary or Fullscreen degraded; both remain first-class when their actual terminal surfaces are available.

## Finite session facts

F1 needs enough authoritative state for later APIs to tell the truth, but not every resolver input belongs on the public object and F1 must not predesign pointer, focus, or input-routing operations. The selected public facts implied by this matrix are:

- public semantic host: live or synchronous string; deterministic testing presents modeled live facts to the component and remains an external observation host;
- requested terminal mode when the host accepts one;
- effective terminal mode or no acquired terminal mode;
- output destination, update policy, and visual or screen-reader presentation;
- terminal dimensions separately from the effective layout dimensions and row bound;
- stable physical origin, renderer-owned element-hit-testing availability, and suspension support;
- history ownership and final-output semantics derived from the finite mode/output combination.

Stdout TTY, stdin TTY, raw-input support, live-update request, and terminal protocol results remain separate resolver inputs. They are not broad public “interactivity” fields; the F1 session exposes only its selected semantic output or capability results, while F3 owns public input availability. Deterministic testing presents modeled production facts to the component and keeps structured observation on `RenderResult`; the internal observer is not a resolver input.

The removed `interactive` boolean is not replaced by a broad “the application is interactive” fact. Historical F1.3 selected a structured readonly session projection; the bounded Path 1 closure keeps that coherent session private and exposes only `useLayoutWidth()` plus setup-time nullable `useViewportHeight()`. F1.4's live resolver and the narrowly named `liveUpdates` override remain implementation evidence, while input availability and acquisition stay separate. The former types, mappings, lifecycle, and rejected alternatives live in the historical [render-session record](./render-session.md).

## F1.1 conclusions

The source, existing vouched records, accepted inline invariant, and real-PTY comparison determine the target matrix:

1. **Unavailable full-screen request outside screen-reader rendering:** use clean, detectable fallback rather than throw or print a warning. A non-TTY or final-output host acquires no terminal mode. Redirected CLI output stays useful and warnings do not pollute pipelines, while application code can still observe the fallback.
2. **Inline ownership and overflow:** vue-tui's generated Inline controls and geometry-safe coordinated TTY writes never clear terminal history or shell output from before the app started. Before the first visible managed output it establishes a fresh row, so even a partially occupied pre-mount row remains outside its erase boundary. Visual layout uses terminal rows as a maximum, recalculates only naturally over-height trees within that allocation, and hard-clips columns plus rows from layout row zero. Static, coordinated safe output, and resize create explicit history boundaries; an old region whose physical rows can no longer be proven remains an immutable snapshot. Tail-follow and selection-centered presentation belong to ScrollBox/application state rather than a global crop. The destructive escape hatch is explicit: use the returned raw streams, clear before mount or after teardown, or use Fullscreen. There is no mounted destructive policy on the coordinated path. Direct process writes cannot silently become part of the guarantee.
3. **Screen reader with a full-screen request:** use the main-screen transcript, do not enter the alternate screen, and do not arm targeted mouse. F1.4 removes the prior hybrid, whose alternate-screen lifetime provided neither full-screen geometry nor durable transcript behavior and could suppress native mouse interaction for no usable event.
4. **Suspend/resume:** external `SIGTSTP`/`SIGCONT` uses restore-before-stop and re-establish-after-continue behavior. Whether Ctrl+Z is a default framework shortcut belongs to F3 input routing; F1 only defines behavior once the process is suspending or continuing.
5. **Default exit output:** inline and transcript leave their output on the main screen; full-screen restores the previous screen and prints no automatic final frame. Applications that need a shell-visible summary produce one explicitly after the session exits.
6. **Fatal error output:** a durable error survives restoration rather than following the clean-exit “no final frame” rule. Inline can retain ErrorOverview; full-screen restores first and then writes the error to stderr, while final-output hosts write it directly.

The Inline ownership guarantee and application-side escape-hatch requirement were explicitly accepted by @hyfdev on 2026-07-11. Bounding mutable output to terminal-addressable rows is a technical consequence of preserving history while retaining replacement semantics; the comparison below demonstrates the available presentations but does not make one clipping policy or escape-hatch API part of the maintainer vouch. The host split, independent input facts, screen-reader fallback, restoration, error durability, static-host behavior, and separate test observations follow from the vouched terminal models and correctness requirements rather than additional product preferences. The clean-slate mount API and non-TTY policy are settled in the unstamped F1.2 section above; the unstamped F1.3 proposal now maps them to public readonly facts.

## Inline overflow PTY comparison

The committed real-PTY comparison runs every policy against a six-row xterm profile with `PRE_APP_HISTORY` written before mount, then performs three reactive revisions and clean teardown. It feeds the byte stream into `@xterm/headless` and inspects both the visible viewport and normal-buffer scrollback rather than inferring the result from ANSI bytes alone.

| Case                                                    | Pre-app history                           | Current state                                                     | Cost or boundary                                                                            |
| ------------------------------------------------------- | ----------------------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| Production full-height Inline                           | Preserved                                 | Latest bounded frame remains; later output starts on a new row    | The live frame may use all terminal rows; teardown advances without resetting history.      |
| Production over-height tree followed by a shorter frame | Preserved                                 | Latest bounded/shorter frame replaces the owned region            | The old ED3 transition is removed; over-height layout is bounded before terminal output.    |
| Application-bounded, head-clipped region                | Preserved                                 | Latest visible head revision replaces earlier revisions           | Generic clipping cannot decide that a transcript should follow its tail.                    |
| Application-selected bounded tail                       | Preserved                                 | Latest tail revision remains visible                              | The application or a semantic viewport chooses follow-tail behavior.                        |
| `Static` plus a small dynamic tail                      | Preserved                                 | Each completed line appears once and only the latest tail remains | This is the honest terminal-history model for completed output.                             |
| Full-screen                                             | Preserved on the restored main screen     | Full-screen frame disappears on exit                              | Use when the application needs arbitrary bounded viewport repaint.                          |
| Explicit pre-mount clear followed by a bounded session  | Deliberately removed before vue-tui mount | Latest bounded frame remains                                      | Demonstrates a session-external low-level escape; vue-tui emits no later destructive clear. |

The durable fixture and assertions are [`inline-overflow-comparison.test.ts`](../../packages/runtime-tests/integration/pty/inline-overflow-comparison.test.ts) and [`inline-overflow-comparison.tsx`](../../packages/runtime-tests/integration/pty/fixtures/inline-overflow-comparison.tsx). The former expected-failure specifications are now ordinary passing regressions.
