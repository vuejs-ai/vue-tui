# Rendering mode and host behavior matrix

> **Status:** completed F1.1 surface model, accepted F1.2 mount and non-TTY contract, completed unstamped F1.3 readonly-session proposal, completed F1.4 private live-mount implementation, completed F1.5 deterministic-test/string-host authority, and completed F1.6 Inline ownership. F1.7 lifecycle completion is Active; public-session publication remains queued. The Inline invariant plus application-side escape-hatch requirement and the finite clean-slate mount behavior were accepted on 2026-07-11; no VOUCHED stamp was added to the later implementation choices.

## Why the matrix has more than two rows

Inline and full-screen are the two live terminal screen models. They are not the only rendering surfaces vue-tui currently produces. A redirected stream, a screen-reader transcript, `renderToString`, and deterministic observation do not acquire the same surface merely because an application or test requested a mode.

The matrix therefore keeps four questions separate:

1. **Render host:** live process, deterministic test, or synchronous string render.
2. **Requested mode:** inline or full-screen, when the host accepts a terminal-mode request.
3. **Effective surface:** main-screen live region, alternate-screen viewport, linear transcript, final stream output, static string, or test observation.
4. **Independent I/O facts:** live-update policy, stdout and stdin TTY state, screen-reader path, and terminal protocol availability.

This matrix is the source behavior model rather than the public type definition. F1.2 settled the mount shape; F1.3 maps these axes into the selected [`useRenderSession()` contract](./render-session.md).

## Current resolution rules

The live renderer now validates and resolves one surface before reserving stdout or mutating terminal state. [`render-session.ts`](../../packages/runtime/src/render-session.ts) owns the pure resolution; [`render.ts`](../../packages/runtime/src/render.ts) consumes its result:

```ts
requestedMode = normalize(options.mode ?? "inline");
liveUpdates = options.liveUpdates ?? (!isInCi && Boolean(stdout.isTTY));
dimensions = resolvePositiveStdoutSizeOrProvenanceAwareTerminalProbe();
surface = resolveLiveSurface({ requestedMode, liveUpdates, stdout, presentation, dimensions });
alternateScreen = surface.kind === "fullscreen-terminal";
targetedMousePath = alternateScreen && stdin.isTTY && terminalSupportsMouse;
```

`mode` accepts only `"inline"`, `"fullscreen"`, or `undefined`; omission becomes an Inline request. Own `fullscreen`, `alternateScreen`, `interactive`, or `debug` keys fail synchronously, including when their value is `undefined`. `liveUpdates` accepts only a boolean or `undefined`. These checks run before stream getters, stdout ownership, terminal probing, raw mode, listeners, Vue setup, or terminal writes. The removed-key error wins when an old key and invalid `mode` appear together.

`liveUpdates` is an output-cadence request, not a statement that the application can receive input. CI and stdout TTY state only choose its default; an explicit value wins. The internal session records requested/effective mode, output, dimensions, and structural capabilities separately. F3 still owns the public raw-input contract; F1.4 temporarily preserves the prior default input-acquisition timing from the resolved live-update request rather than publishing it as a session fact.

Terminal dimensions carry provenance. A complete positive stdout pair wins. If it is incomplete, one complete positive pair from process stdout/stderr, `COLUMNS`/`LINES`, a controlling-terminal ioctl, `tput`, or Linux `resize` replaces it atomically; fields from two sources are never spliced into a claimed viewport. Process-global fallback is used only for the process's own stdout/stderr, not an arbitrary custom TTY. Probe failure returns unavailable instead of manufacturing `80×24`. A valid partial custom field may still inform unbounded layout or the transitional numeric `useWindowSize()` projection, but it cannot acquire a visual terminal mode. The renderer may choose 80 layout columns and 24 transitional rows, but those defaults never establish `dimensions.terminal`.

Explicit `liveUpdates: true` is the route to L2/L3 and overrides the CI default. Explicit `liveUpdates: false` always selects L1 before TTY or size fallbacks are considered.

### Observation and scheduling

Deterministic commit observation uses an internal render observer that receives structured `{ dynamic, staticOutput }` content independently from terminal bytes. Installing the observer does not select a host, change update cadence, append frames to stdout, or manufacture a capability. `maxFps: 0` selects unthrottled commit scheduling only. The former `debug` mount option is removed and fails synchronously before stream inspection or terminal mutation; `@vue-tui/testing` likewise rejects a `debug` render option and exposes observation through `RenderResult` instead.

### Development-reload modifier

An in-place Vue hot update keeps the mounted session and repaints through its existing surface. A Vite full reload runs the registered internal teardown before replacing the app, restoring terminal modes and the alternate screen without settling the old app's exit promise; the newly imported app then acquires its surface again. The target keeps that distinction: a hot update preserves the session, while a full reload must completely release and reacquire terminal ownership.

### Input and hit-map modifier

stdin is independent of the output surface:

| stdin and surface                           | Current behavior                                                                                                                                                                            |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| stdin is non-TTY                            | Raw mode is unsupported. `useInput` and `useMouseInput` can throw while guarded focus behavior degrades; targeted mouse is unavailable.                                                     |
| stdin is TTY, stdout is non-TTY             | Input composables can acquire raw input even in L1-L3. Mouse and paste reporting cannot write their terminal control modes to the non-TTY stdout.                                           |
| stdin and stdout are TTY, surface is not L6 | Terminal-wide input and `useMouseInput` can work, but targeted element mouse is unavailable. Inline mouse reporting still takes native selection and wheel behavior away from the terminal. |
| stdin and stdout are TTY, surface is L6     | The current normal visual Fullscreen path can align terminal mouse coordinates with the clipped element hit map when the terminal supports SGR mouse.                                       |
| stdin and stdout are TTY, surface is L7     | Keyboard/raw input can remain available, but Fullscreen-targeted mouse is not armed and the renderer reports no element hit-testing capability for the transcript.                          |

## F1.2 accepted mount contract

F1.2 settles one canonical `createApp` and one optional public mode field, `mode: "inline" | "fullscreen"`. Omission is an Inline request for convenience; it does not make Inline the primary product mode or Fullscreen a degraded path. The old mode booleans are removed directly because the project is experimental; F1.4 also replaces the broad `interactive` output override with `liveUpdates`.

On a TTY, an explicit live-update override also overrides the CI default and resolves through the corresponding TTY row when terminal dimensions are usable. The live resolver first accepts a positive stdout size, then tries the provenance-aware controlling-terminal probe. A positive detected width may still supply layout width, and 80 is the final layout-column default, but neither 80 nor the transitional 24-row default is labeled a detected terminal without a positive sourced pair. Visual mode acquisition without that pair falls back to final stream output. The override changes output cadence, not terminal capabilities: it cannot acquire Fullscreen, an alternate screen, stable viewport, or element hit testing on a non-TTY stream. Deterministic observation is supplied separately by the internal observer and therefore cannot change any row. Likewise, stdin and raw-input behavior remain independent resolver inputs rather than consequences of the public output facts.

The test host exposes structured content frames and an emulated final terminal surface as different observations. A final-stream test uses the production final-output path, while an explicitly live stream test uses the production updater without claiming a terminal mode. Static and test dimensions are supplied deliberately by their hosts and never borrowed from the developer's process streams.

Fallback only describes an environment in which the requested screen model cannot exist. It does not make Inline primary or Fullscreen degraded; both remain first-class when their actual terminal surfaces are available.

## Finite session facts

F1 needs enough authoritative state for later APIs to tell the truth, but not every resolver input belongs on the public object and F1 must not predesign pointer, focus, or input-routing operations. The selected public facts implied by this matrix are:

- public semantic host: live or synchronous string; deterministic testing presents modeled live facts to the component and remains an external observation host;
- requested terminal mode when the host accepts one;
- effective terminal mode or no acquired terminal mode;
- output destination, update policy, and visual or screen-reader presentation;
- terminal dimensions separately from the effective layout dimensions and row bound;
- stable physical origin, renderer-owned element-hit-testing availability, and suspension support;
- history ownership and final-output semantics derived from the finite mode/output combination.

Stdout TTY, stdin TTY, raw-input support, live-update request, and terminal protocol results remain separate resolver inputs. They are not broad public “interactivity” fields; the F1 session exposes only its selected semantic output or capability results, while F3 owns public input availability. Deterministic testing presents modeled production facts to the component and keeps structured observation on `RenderResult`; the internal observer is not a resolver input.

The removed `interactive` boolean is not replaced by a broad “the application is interactive” fact. F1.3 selects one structured readonly reactive session with small discriminated mode/output values, semantic structural capabilities, and distinct terminal-versus-layout dimensions. F1.4 implements its live source and the narrowly named `liveUpdates` override; F3 still owns input availability and acquisition. Exact types, the derived `useLayoutSize()` projection, mappings, lifecycle, current-API dispositions, and rejected alternatives live in [render-session.md](./render-session.md).

## F1.1 conclusions

The source, existing vouched records, accepted inline invariant, and real-PTY comparison determine the target matrix:

1. **Unavailable full-screen request outside screen-reader rendering:** use clean, detectable fallback rather than throw or print a warning. A non-TTY or final-output host acquires no terminal mode. Redirected CLI output stays useful and warnings do not pollute pipelines, while application code can still observe the fallback.
2. **Inline ownership and overflow:** vue-tui's generated Inline controls and geometry-safe coordinated TTY writes never clear terminal history or shell output from before the app started. Before the first visible managed output it establishes a fresh row, so even a partially occupied pre-mount row remains outside its erase boundary. Visual layout uses terminal rows as a maximum, recalculates only naturally over-height trees within that allocation, and hard-clips columns plus rows from layout row zero. Static, coordinated safe output, and resize create explicit history boundaries; an old region whose physical rows can no longer be proven remains an immutable snapshot. Tail-follow and selection-centered presentation belong to ScrollBox/application state rather than a global crop. The destructive escape hatch is explicit: use the returned raw streams, clear before mount or after teardown, or use Fullscreen. There is no mounted destructive policy on the coordinated path. Direct process writes cannot silently become part of the guarantee.
3. **Screen reader with a full-screen request:** use the main-screen transcript, do not enter the alternate screen, and do not arm targeted mouse. F1.4 removes the prior hybrid, whose alternate-screen lifetime provided neither full-screen geometry nor durable transcript behavior and could suppress native mouse interaction for no usable event.
4. **Suspend/resume:** external `SIGTSTP`/`SIGCONT` uses restore-before-stop and re-establish-after-continue behavior. Whether Ctrl+Z is a default framework shortcut belongs to F3 input routing; F1 only defines behavior once the process is suspending or continuing.
5. **Default exit output:** inline and transcript leave their output on the main screen; full-screen restores the previous screen and prints no automatic final frame. Applications that need a shell-visible summary produce one explicitly after the session exits.
6. **Fatal error output:** a durable error survives restoration rather than following the clean-exit “no final frame” rule. Inline can retain ErrorOverview; full-screen restores first and then writes the error to stderr, while final-output hosts write it directly.

The Inline ownership guarantee and application-side escape-hatch requirement were explicitly accepted by @hyf0 on 2026-07-11. Bounding mutable output to terminal-addressable rows is a technical consequence of preserving history while retaining replacement semantics; the comparison below demonstrates the available presentations but does not make one clipping policy or escape-hatch API part of the maintainer vouch. The host split, independent input facts, screen-reader fallback, restoration, error durability, static-host behavior, and separate test observations follow from the vouched terminal models and correctness requirements rather than additional product preferences. The clean-slate mount API and non-TTY policy are settled in the unstamped F1.2 section above; the unstamped F1.3 proposal now maps them to public readonly facts.

## Inline overflow PTY comparison

The committed real-PTY comparison runs every policy against a six-row xterm profile with `PRE_APP_HISTORY` written before mount, then performs three reactive revisions and clean teardown. It feeds the byte stream into `@xterm/headless` and inspects both the visible viewport and normal-buffer scrollback rather than inferring the result from ANSI bytes alone.

| Case                                                    | Pre-app history                           | Current state                                                     | Cost or boundary                                                                            |
| ------------------------------------------------------- | ----------------------------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| Production full-height Inline                           | Preserved                                 | Latest bounded frame remains; later output starts on a new row    | The live frame may use all terminal rows; teardown advances without resetting history.      |
| Production over-height tree followed by a shorter frame | Preserved                                 | Latest bounded/shorter frame replaces the owned region            | The old ED3 transition is removed; over-height layout is bounded before terminal output.    |
| Application-bounded, head-clipped region                | Preserved                                 | Latest visible head revision replaces earlier revisions           | Generic clipping cannot decide that a transcript should follow its tail.                    |
| Application-selected bounded tail                       | Preserved                                 | Latest tail revision remains visible                              | The application or a semantic viewport chooses follow-tail behavior.                        |
| `Static` plus a small dynamic tail                      | Preserved                                 | Each completed line appears once and only the latest tail remains | This is the honest terminal-history model for completed output.                             |
| Full-screen                                             | Preserved on the restored main screen     | Full-screen frame disappears on exit                              | Use when the application needs arbitrary bounded viewport repaint.                          |
| Explicit pre-mount clear followed by a bounded session  | Deliberately removed before vue-tui mount | Latest bounded frame remains                                      | Demonstrates a session-external low-level escape; vue-tui emits no later destructive clear. |

The durable fixture and assertions are [`inline-overflow-comparison.test.ts`](../../packages/runtime-tests/integration/pty/inline-overflow-comparison.test.ts) and [`inline-overflow-comparison.tsx`](../../packages/runtime-tests/integration/pty/fixtures/inline-overflow-comparison.tsx). The former expected-failure specifications are now ordinary passing regressions. [`inline-resize-history.test.ts`](../../packages/runtime-tests/integration/pty/inline-resize-history.test.ts) adds chronological real-PTY/xterm resize evidence for both visual Inline and the screen-reader transcript.
