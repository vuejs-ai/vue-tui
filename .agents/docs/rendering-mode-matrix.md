# Rendering mode and host behavior matrix

> **Status:** completed F1.1 surface model, accepted F1.2 mount and non-TTY contract, and completed unstamped F1.3 readonly-session proposal. This record describes the implementation at `f1ce02b`, the target conclusions supported by the terminal models, the Inline invariant plus application-side escape-hatch requirement accepted on 2026-07-11, and the finite clean-slate mount behavior accepted later that day. The selected public session shape is recorded separately; neither it nor the mount replacement and remaining F1 behavior is implemented.

## Why the matrix has more than two rows

Inline and full-screen are the two live terminal screen models. They are not the only rendering surfaces vue-tui currently produces. A redirected stream, a screen-reader transcript, `renderToString`, and the content-frame testing harness do not acquire the same surface merely because the runtime internally falls through a boolean called `fullscreen`.

The matrix therefore keeps four questions separate:

1. **Render host:** live process, deterministic test, or synchronous string render.
2. **Requested mode:** inline or full-screen, when the host accepts a terminal-mode request.
3. **Effective surface:** main-screen live region, alternate-screen viewport, linear transcript, final stream output, static string, or test observation.
4. **Independent I/O facts:** live-update policy, stdout and stdin TTY state, screen-reader path, and terminal protocol availability.

This matrix is the source behavior model rather than the public type definition. F1.2 settled the mount shape; F1.3 maps these axes into the selected [`useRenderSession()` contract](./render-session.md).

## Current resolution rules

The live renderer currently resolves its main branches as follows in [`render.ts`](../../packages/runtime/src/render.ts):

```ts
requestedFullscreen = Boolean(options.fullscreen ?? options.alternateScreen);
liveUpdates = options.interactive ?? (!isInCi && Boolean(stdout.isTTY));
alternateScreen = requestedFullscreen && liveUpdates && Boolean(stdout.isTTY);
fixedViewport = alternateScreen && !isScreenReaderEnabled;
targetedMousePath = alternateScreen && stdin.isTTY && terminalSupportsMouse;
```

`liveUpdates` is the truthful name for what the current `interactive` option primarily controls. It selects the live writer, resize listener, signal handler, Kitty initialization, and default lifetime raw-mode hold. It is not a complete statement that the application can or cannot receive input: current input composables can independently acquire raw mode on a TTY stdin even when `interactive: false`.

CI only changes the default for `interactive`; an explicit value wins. `fullscreen` wins over the deprecated `alternateScreen` alias whenever `fullscreen` is not `null` or `undefined`, including `fullscreen: false` with `alternateScreen: true`. An unavailable full-screen request is currently ignored without an error, warning, or observable requested-versus-effective fact.

## Current live-host matrix

These rows are exhaustive for the axes that select a rendering surface. `debug` and stdin behavior are finite modifiers covered immediately afterward.

| ID  | Requested mode | Current `interactive` result | stdout  | Screen reader | Current effective surface                                                                                                                                                                 |
| --- | -------------- | ---------------------------- | ------- | ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| L1  | Either         | `false`                      | Either  | Either        | **Final-output stream.** No live terminal surface; new `Static` output is written as it appears and only the latest dynamic frame is written at teardown.                                 |
| L2  | Either         | `true`                       | Non-TTY | Off           | **Forced relative stream updater.** The inline writer emits updates to a stream, but there is no terminal screen, reliable physical origin, or hit map. A full-screen request is ignored. |
| L3  | Either         | `true`                       | Non-TTY | On            | **Forced transcript updater.** Linear screen-reader output is updated on the stream; a full-screen request is ignored.                                                                    |
| L4  | Inline         | `true`                       | TTY     | Off           | **Inline live region.** Relative updates on the main screen; completed output may move into terminal-owned history.                                                                       |
| L5  | Full-screen    | `true`                       | TTY     | Off           | **Fixed full-screen viewport.** Alternate screen, terminal-sized layout, origin `(0, 0)`, clipping, complete repaint, and a reliable hit-map coordinate system.                           |
| L6  | Inline         | `true`                       | TTY     | On            | **Main-screen transcript.** Linearized output with no fixed height, stable physical origin, or hit map.                                                                                   |
| L7  | Full-screen    | `true`                       | TTY     | On            | **Current hybrid defect.** The runtime enters the alternate screen and hides the cursor, but renders an unbounded linear transcript and builds no usable hit map.                         |

The normal default maps a non-CI TTY to L4-L7 according to the requested mode and screen-reader state, and a non-TTY or CI environment to L1. Forced `interactive: true` is the only route to L2 or L3.

### Debug modifier

`debug` does not participate in mode resolution, but it changes observation and output behavior:

| Base case    | Current debug behavior                                                                                                                                                                                                                                                 |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| L5           | The fixed viewport still clears, homes, and replaces the alternate-screen surface. The internal test frame sink also receives the content frame.                                                                                                                       |
| L1-L4, L6-L7 | Every commit appends the complete current content instead of maintaining the normal relative live region. In screen-reader cases the content is linearized; full `Static` history is replayed on later commits. L7 still enters the alternate screen before appending. |

Debug is an implementation and observation switch, not a third product rendering mode. The target contract must not require application code to infer terminal semantics from it.

### Development-reload modifier

An in-place Vue hot update keeps the mounted session and repaints through its existing surface. A Vite full reload runs the registered internal teardown before replacing the app, restoring terminal modes and the alternate screen without settling the old app's exit promise; the newly imported app then acquires its surface again. The target keeps that distinction: a hot update preserves the session, while a full reload must completely release and reacquire terminal ownership.

### Input and hit-map modifier

stdin is independent of the output surface:

| stdin and surface                           | Current behavior                                                                                                                                                                                                                    |
| ------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| stdin is non-TTY                            | Raw mode is unsupported. `useInput` and `useMouseInput` can throw while guarded focus behavior degrades; targeted mouse is unavailable.                                                                                             |
| stdin is TTY, stdout is non-TTY             | Input composables can acquire raw input even in L1-L3. Mouse and paste reporting cannot write their terminal control modes to the non-TTY stdout.                                                                                   |
| stdin and stdout are TTY, surface is not L5 | Terminal-wide input and `useMouseInput` can work, but there is no reliable element hit map. Inline mouse reporting still takes native selection and wheel behavior away from the terminal.                                          |
| stdin and stdout are TTY, surface is L5     | The current normal visual full-screen path can align terminal mouse coordinates with the clipped element hit map when the terminal supports SGR mouse.                                                                              |
| stdin and stdout are TTY, surface is L7     | Existing element handlers can arm SGR mouse reporting, but the screen-reader renderer produces no hit map, so the handlers cannot receive a target. This can suppress native mouse behavior without providing application behavior. |

Interactive mounts also restore on SIGINT, SIGTERM, and SIGHUP. There is no suspend/resume implementation. A second live app on the same stdout is warned and ignored; direct process-stream writes and process-global console patching remain outside reliable multi-app coordination.

## F1.2 accepted mount contract

F1.2 settles one canonical `createApp` and one optional public mount field, `mode: "inline" | "fullscreen"`. Omission is an Inline request for convenience; it does not make Inline the primary product mode or Fullscreen a degraded path. The old booleans are removed directly in the target because the project is experimental.

### Input and validation

| Runtime input                                                                                                              | Normalized requested mode | Result                                                   |
| -------------------------------------------------------------------------------------------------------------------------- | ------------------------- | -------------------------------------------------------- |
| `mode` omitted                                                                                                             | Inline                    | Accept                                                   |
| `mode: undefined`                                                                                                          | Inline                    | Accept as omission                                       |
| `mode: "inline"`                                                                                                           | Inline                    | Accept                                                   |
| `mode: "fullscreen"`                                                                                                       | Full-screen               | Accept                                                   |
| Any other `mode` value, including `null`, booleans, other strings, numbers, objects, arrays, functions, symbols, or bigint | None                      | Throw a clear programming error synchronously            |
| Own `fullscreen` property, including `fullscreen: undefined`                                                               | None                      | Throw a clear removed-option error synchronously         |
| Own `alternateScreen` property, including `alternateScreen: undefined`                                                     | None                      | Throw a clear removed-option error synchronously         |
| Either removed property together with any `mode` value                                                                     | None                      | Throw the removed-option error before considering `mode` |

The implementation must detect removed keys with own-property semantics and validate before stream reservation, listeners, console patching, raw mode, cursor output, alternate-screen entry, or Vue mount. Unknown unrelated JavaScript keys are outside F1.2; this checkpoint does not invent generic strict-object validation. After validation and defaulting, every live mount records exactly one normalized requested mode.

### Accepted host matrix

The target keeps Inline and Fullscreen as the only live terminal modes while refusing to call every other output surface Inline. “No effective mode” means that no live terminal screen model was acquired; the application may still produce useful stream output. The current `interactive` field demonstrates the explicit live-update override, but its final name and target disposition remain later F1 work.

On a TTY, an explicit live-update override also overrides the CI default and resolves through the corresponding TTY row when terminal dimensions are usable. The live resolver first accepts a positive stdout size, then tries the pinned-Ink controlling-terminal probe. A positive probed width may still supply layout width, and 80 is the final layout-column default, but neither fallback is labeled a detected terminal unless a positive terminal pair was resolved. Visual mode acquisition without that pair falls back to final stream output. The override changes output cadence, not terminal capabilities: it cannot acquire Fullscreen, an alternate screen, stable viewport, or element hit testing on a non-TTY stream. `debug: true` is a separate diagnostic path that appends complete current output on commits and adds a trailing newline at non-interactive teardown; it does not change the mode resolution above or acquire capabilities by itself. Likewise, stdin and raw-input behavior remain independent resolver inputs rather than consequences of this stdout policy.

The test host should expose content frames and an emulated final terminal surface as different observations. A final-output test must reproduce the production final-stream class rather than combining `debug: true` with a misleading `interactive: false`. Static and test virtual dimensions must be supplied by their host rather than leaking the developer's real process streams.

Fallback only describes an environment in which the requested screen model cannot exist. It does not make Inline primary or Fullscreen degraded; both remain first-class when their actual terminal surfaces are available.

## Target lifecycle contract

| Surface                    | Target promise                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Inline live region         | Never erase terminal history or shell output that existed before the app. Limit replaceable output to rows the terminal can still address; once output enters terminal-owned history, leave it immutable. A terminal-history commit operation—currently provided by `Static`—can explicitly transfer completed output to that history, but the final API placement remains open. The exact non-destructive presentation for a longer dynamic layout—root clamping, application-selected head or tail projection, a bounded viewport, or another mechanism—remains implementation and API work. When resize or geometry change makes old live rows unreachable, leave that old snapshot immutable and establish a new bounded live region at the bottom. Coordinated output may move the live region but must re-establish it without claiming a stable physical origin. Applications can choose full-screen for arbitrary viewport repaint and must retain an explicit escape hatch when they intentionally want destructive main-screen behavior. | Main-screen transcript     | Remain linear and main-screen based. ## Finite session facts

F1 needs enough authoritative state for later APIs to tell the truth, but not every resolver input belongs on the public object and F1 must not predesign pointer, focus, or input-routing operations. The selected public facts implied by this matrix are:

- public semantic host: live or synchronous string; deterministic testing presents modeled live facts to the component and remains an external observation host;
- requested terminal mode when the host accepts one;
- effective terminal mode or no acquired terminal mode;
- output destination, update policy, and visual or screen-reader presentation;
- terminal dimensions separately from the effective layout dimensions and row bound;
- stable physical origin, renderer-owned element-hit-testing availability, and suspension support;
- history ownership and final-output semantics derived from the finite mode/output combination.

Stdout TTY, stdin TTY, raw-input support, debug observation, live-update request, and terminal protocol results remain separate resolver inputs. They are not broad public “interactivity” fields; the F1 session exposes only its selected semantic output or capability results, while F3 owns public input availability. Deterministic testing presents modeled production facts to the component and keeps test-only observation on `RenderResult`.

The current `interactive` boolean must not be published as a broad “the application is interactive” fact: it does not gate all input composables today. F1.3 selects one structured readonly reactive session with small discriminated mode/output values, semantic structural capabilities, and distinct terminal-versus-layout dimensions. Exact types, the derived `useLayoutSize()` projection, mappings, lifecycle, current-API dispositions, and rejected alternatives live in [render-session.md](./render-session.md).

## F1.1 conclusions

The source, existing vouched records, accepted inline invariant, and real-PTY comparison determine the target matrix:

1. **Unavailable full-screen request outside screen-reader rendering:** use clean, detectable fallback rather than throw or print a warning. A non-TTY or final-output host acquires no terminal mode. Redirected CLI output stays useful and warnings do not pollute pipelines, while application code can still observe the fallback.
2. **Inline ownership and overflow:** vue-tui's coordinated inline renderer never clears terminal history or shell output from before the app started. Replaceable output cannot exceed rows the terminal can still address; rows that enter terminal-owned history become immutable. Append-only snapshots preserve history but no longer provide replacement semantics. The comparison demonstrates distinct non-destructive presentations through the current `Static` mechanism, a bounded head or tail view, and full-screen, but it does not select the final history API, exact root clamping, or head-versus-tail behavior. When old live rows become unreachable, they remain an immutable snapshot and the runtime establishes a new bounded region. Applications must retain an explicit escape hatch when they intentionally want destructive main-screen behavior. The comparison proves pre-mount terminal ownership as one viable low-level form; the supported escape-hatch API and whether it also permits a runtime-coordinated active-session reset remain open. Direct process writes cannot silently become part of the guarantee.
3. **Screen reader with a full-screen request:** use the main-screen transcript, do not enter the alternate screen, and do not arm targeted mouse. The current hybrid provides neither full-screen geometry nor durable transcript behavior and can suppress native mouse interaction for no usable event.
4. **Suspend/resume:** external `SIGTSTP`/`SIGCONT` uses restore-before-stop and re-establish-after-continue behavior. Whether Ctrl+Z is a default framework shortcut belongs to F3 input routing; F1 only defines behavior once the process is suspending or continuing.
5. **Default exit output:** inline and transcript leave their output on the main screen; full-screen restores the previous screen and prints no automatic final frame. Applications that need a shell-visible summary produce one explicitly after the session exits.
6. **Fatal error output:** a durable error survives restoration rather than following the clean-exit “no final frame” rule. Inline can retain ErrorOverview; full-screen restores first and then writes the error to stderr, while final-output hosts write it directly.

The Inline ownership guarantee and application-side escape-hatch requirement were explicitly accepted by @hyf0 on 2026-07-11. Bounding mutable output to terminal-addressable rows is a technical consequence of preserving history while retaining replacement semantics; the comparison below demonstrates the available presentations but does not make one clipping policy or escape-hatch API part of the maintainer vouch. The host split, independent input facts, screen-reader fallback, restoration, error durability, static-host behavior, and separate test observations follow from the vouched terminal models and correctness requirements rather than additional product preferences. The clean-slate mount API and non-TTY policy are settled in the unstamped F1.2 section above; the unstamped F1.3 proposal now maps them to public readonly facts.

## Inline overflow PTY comparison

The committed real-PTY comparison runs every policy against a six-row xterm profile with `PRE_APP_HISTORY` written before mount, then performs three reactive revisions and clean teardown. It feeds the byte stream into `@xterm/headless` and inspects both the visible viewport and normal-buffer scrollback rather than inferring the result from ANSI bytes alone.

| Case                                                   | Pre-app history                           | Current state                                                     | Cost or boundary                                                                                   |
| ------------------------------------------------------ | ----------------------------------------- | ----------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| Current full-height inline                             | Lost on clean teardown                    | Final frame remains                                               | Teardown emits `ED2 + ED3 + Home`; `ED3` deletes scrollback.                                       |
| Current overflow followed by a shorter frame           | Lost during the first shorter update      | Shorter frame remains                                             | The transition clears before teardown, so fixing only final cleanup is insufficient.               |
| Application-bounded, head-clipped region               | Preserved                                 | Latest visible head revision replaces earlier revisions           | Generic clipping cannot decide that a transcript should follow its tail.                           |
| Application-selected bounded tail                      | Preserved                                 | Latest tail revision remains visible                              | The application or a semantic viewport chooses follow-tail behavior.                               |
| `Static` plus a small dynamic tail                     | Preserved                                 | Each completed line appears once and only the latest tail remains | This is the honest terminal-history model for completed output.                                    |
| Append-only complete frames                            | Preserved                                 | Every revision remains                                            | Spinner, progress, and token updates create duplicate snapshots rather than replacement semantics. |
| Full-screen                                            | Preserved on the restored main screen     | Full-screen frame disappears on exit                              | Use when the application needs arbitrary bounded viewport repaint.                                 |
