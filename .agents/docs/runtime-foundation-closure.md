## Predeclared acceptance bounds

These bounds were chosen before running the committed harnesses.

### Correctness and ordering

- Every deterministic assertion and packed-consumer assertion must pass exactly; snapshots cannot be weakened to accept current output.
- Inline emits no framework-generated ED2, ED3, or Home, never rewrites pre-app history, and preserves once-only history order across resize, coordinated output, suspension, continuation, normal teardown, and a documented fatal path.
- Fullscreen keeps one fixed successful-frame authority for visible cells, geometry, caret, pointer, and selection and restores every acquired mode on every exit path.
- A history item becomes committed when its complete output transaction's `write()` call returns normally. A `false` return is accepted backpressure and counts as committed. A throw is an indeterminate fatal handoff and is never retried automatically. Paint, observation, or pre-handoff failure must not advance the history cursor.
- The accepted collection signature defines positional acceptance identity explicitly: after an item is committed, every position in that prefix must remain `Object.is`-identical. Replacement arrays and changes in the uncommitted tail are allowed; shrinking, replacing, or reordering a committed position fails clearly; mutation through the same object identity does not rewrite emitted history; remounting begins a new region.

### Responsiveness

- These budgets are user-visible failure guards for the fixed product-sized journeys, not a requirement to make small work scale only with its affected area. The 500-record transcript, 2,000-record finder model, 500-line document, 120-row monitor, and four-pane workbench deliberately exceed the ordinary initial examples while remaining bounded. Passing them accepts the current full-render architecture even when internal work still scales with the mounted surface.
- Input-to-visible latency has p95 no greater than 200 ms and no action greater than 500 ms for J2 through J5. The 200 ms outer line is the direct-feedback requirement for filtering, navigation, approval, scrolling, selection, and pane switching in the recorded interactive scenarios; the 500 ms line prevents a visibly stalled action from disappearing inside a percentile. If p95 is greater than 100 ms but no greater than 200 ms, the reasonable immediate-feedback alternatives could change the conclusion, so return to Yunfei with the measurements and the cost of meeting 100 ms before accepting the architecture.
- A 20 ms event-loop heartbeat has p99 excess delay no greater than 100 ms and no excess delay greater than 200 ms during each workload, so framework work cannot consume the complete direct-feedback budget before an input callback can run.
- The latency sample population is every scripted action-to-observed-screen transition across the ten measured repetitions: query edits and navigation in J2, scroll/selection/copy in J3, metric mutations and quit in J4, and focus/wheel/drag/overlay actions in J5. Warmups are excluded; p95 and p99 use the nearest-rank estimator over the pooled samples for one journey. Render/paint duration is recorded separately for diagnosis but has no independent architecture gate.
- Wall-clock values are local acceptance evidence and are recorded with hardware, Node, repetition count, raw samples, and estimator. CI checks the harness structure and deterministic work counts but no timing threshold.

### Resource and backpressure

- Runtime-owned pending output is bounded independently of producer volume: at most one transaction handed to the stream, no further stream write after `write()` returns `false` until `drain`, and at most one latest replaceable desired frame per application. New declarative history remains in application-owned Vue state until it can be prepared and committed; Runtime may not copy each blocked generation into a byte queue.
- A coordinated-output call made while the stream cannot accept another transaction must have an explicit caller-visible non-acceptance or flow-control result and must not be retained in an unbounded Runtime queue. The current synchronous `void` shape is evidence, not an exemption; if J6 demonstrates that it cannot satisfy this invariant, the public alternatives require the Stage 3 maintainer-decision process.
- On a normal writable, output order is FIFO and `waitUntilRenderFlush()` does not resolve until all accepted bytes for the requested generation have reached the stream callback barrier.
- After the first `write()` that returns `false`, Runtime may not call `write()` again until `drain`. The stream's `writableLength` may exceed its 256-byte high-water mark only by the largest single atomic output transaction already handed to it.
- After the finite producer stops, queued non-coalescible records drain in order, the latest dynamic frame becomes visible, all queue and listener state returns to zero, and teardown restores the terminal.
- Heap and RSS are recorded after exposed GC for every warmup and measured repetition but are not noisy absolute architecture gates. Repeat each fixed workload ten times in one process plus a matched mount/unmount no-op control.
